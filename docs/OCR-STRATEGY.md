# 图片 OCR 策略

## 决策

默认不做全量 OCR。未来提供三种模式：

| 模式 | 行为 | 适合场景 |
| --- | --- | --- |
| `off` | 不识别图片文字 | 默认；忠实归档原文，零额外噪声 |
| `auto` | 只识别信息主要存在于图片中的内容 | 推荐；兼顾可搜索性和正文质量 |
| `all` | 尝试识别全部有效图片 | 专门的数据抽取或无障碍处理 |

公众号图片中有大量二维码、宣传海报、品牌口号、装饰性标题和正文重复截图。全量 OCR 会增加运行时间、文件体积和误识别，还会污染全文搜索。因此，“能识别”不是“应该写入 Markdown”的充分条件。

## `auto` 的建议选择规则

满足以下任一条件时进入 OCR 候选集：

- 微信消息类型是图片分享，正文主要由图片组成。
- 去除标题后的正文极短，但有多张大图。
- 用户通过文章、日期范围或图片序号明确选择。
- 图片有较高文字占比，且识别结果与正文不重复。

以下图片默认跳过：

- 二维码、头像、分隔线、表情和小尺寸装饰图。
- 内容哈希重复的图片。
- 明显是推广海报，且文章正文已包含相同信息。
- 平均置信度低于阈值的结果。

“推广海报”不应只靠关键词硬判。可组合二维码检测、版面文字密度、联系方式/价格模式和正文重复率，并保留人工覆盖入口。

## 输出原则

1. 图片原文件始终是事实来源，OCR 只是派生数据。
2. 每张候选图片先写 sidecar，例如 `images/.../001.jpg.ocr.json`，保存识别文本、置信度、语言、provider 和时间。
3. 仅当 `auto` 判定有语义价值且置信度达标时，才在对应图片后加入折叠块：

```html
<details>
<summary>图片文字（OCR 自动识别，可能有误）</summary>

识别出的文字……

</details>
```

4. frontmatter 只增加可选字段，例如 `ocr_status`、`ocr_image_count` 和 `ocr_provider`；不开启 OCR 时不改变现有文件。
5. `verify` 校验 sidecar 与图片一一对应，但 OCR 失败不应让原始文章归档失败。

## 实现落点

新增 `src/enrichers/ocr.mjs`，输入本地图片清单与正文统计，输出结构化 OCR 结果；`src/markdown.mjs` 只负责按策略插入结果。provider 接口保持很小：

```js
async function recognize(imagePath) {
  return { text, confidence, language, blocks };
}
```

macOS 可先接 Apple Vision，本地跨平台可接 Tesseract，云服务作为显式选择。任何 provider 都不得在未告知用户时上传图片。
