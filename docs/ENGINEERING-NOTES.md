# 从“能下载”到“能证明”：工程复盘

## 最重要的认识

归档工具最难的部分不是 HTML 转 Markdown，而是回答“怎么证明没有漏”。如果只看下载命令有没有报错，会忽略列表分页、同名账号、空正文、特殊消息、丢图和错误状态分类。

因此 MPVault 从一开始就把检查点、manifest 和独立 verify 当作核心产品能力，而不是下载结束后的附加统计。

## 真实遇到的兼容性问题

### 名称相同不等于账号相同

微信搜索结果中的可见 alias 不能始终代表原始 ID。可靠做法是搜索候选，再打开候选文章，从页面中的 `user_name` 核验 `gh_...`。页面同时存在普通字符串和 `JsDecode(...)` 两种写法。

### 列表总数与文章数不是同一个概念

接口分页按群发消息前进，一次群发可能包含多篇 `appmsgex`。如果用文章条数推进 `begin`，会跳页或重复。正确做法是按 `itemidx = 1` 的消息数推进，同时把所有子文章放入稳定去重集合。

### 有 `#js_content` 也可能没有正文

某些分享页面保留空正文容器，真正内容放在 `window.cgiDataNew`。应先检查节点是否有文本或媒体；为空时解析 `content_noencode`。图片分享消息还要解析 `picture_page_info_list`。

### 解析失败不能记作文章不可用

只有页面明确显示删除、违规或暂不可用提示时，才能进入 terminal 状态。找不到正文、脚本结构变化或下载异常都必须标为 `failed`，让续传重试并让 verify 失败。

### 网络重试和业务状态要分开

指数退避适合恢复瞬时网络问题，但明确删除/违规不应重试。区分两者，才能既不浪费请求，也不永久漏掉可恢复文章。

## 下次直接复用的检查单

1. 先确认登录账号类型与会话安全。
2. 用原始 ID 核验目标，不用名称或头像猜。
3. 先跑一页，确认分页单位和文章唯一键。
4. 每页保存检查点，限流时保留目录原地续传。
5. 把特殊消息和空正文当作一等测试场景。
6. 只有显式微信错误才归类为不可用；未知情况保持失败。
7. 同时核对 manifest 状态、Markdown 正文和本地图片引用。
8. enrichment 与原始归档分离，不让 OCR、统计或 AI 失败污染事实来源。
